Page Composition Pattern

All pages follow a consistent server/client split:

  • Server component (page.tsx): Exports static metadata, renders layout shell or delegates to a client component.
  • Client components: Handle all interactivity — forms, menus, animations, maps.
  • RevealObserver is placed once per page to power scroll animations across all .reveal elements.

Data Flow

data/hiringStatus.ts  →  careers/page.tsx  →  BranchHiringCard + ApplicationForm
lib/email.ts          →  all api/*/route.ts routes
lib/emailTemplate.ts  →  all api/*/route.ts routes

Page Structure

Home (/) Server

  1. Hero — Full-viewport background image, large H1, orange accent lines, CTA
  2. Services — 4-card grid (Roadworks, Events, TMP Design, Training) with CSS background images

  1. Hero — Eyebrow + H1 + intro paragraph
  2. Our Beginnings — 2-col: origin story + founders SVG
  3. Timeline — 9 company milestones (2008–2025), alternating desktop layout
  4. Locations — Interactive SVG NZ map with 5 clickable branch pins
  5. Leadership Grid — 8 team members in 4-col responsive grid with photos
  6. MW Group — 2×2 card grid: 4 group entities with logos and external links
  7. CTA — Full-width orange stripe CTA

Careers (/careers) Server

  1. Hero — Large H1 + intro + two CTA buttons
  2. Before You Apply — 4 requirements (Right to Work, Drug Test, Licence, Criminal Check)
  3. Why Work With Us — 6 benefit cards in 2-col grid with Lucide icons
  4. Career Path — 6 roles in 2-row visual progression (TTM Worker → Manager)
  5. Current Openings — Branch hiring cards sorted: hiring → limited → closed
  6. Application Form — Full 2-step form wrapped in ApplicationSection

Data-driven: Branch hiring cards populated from data/hiringStatus.ts.

Contact (/contact) Hybrid

  1. Hero — Heading + phone/email contact alternatives
  2. Estimate Info Box — Orange-tinted callout linking to /estimate
  3. Contact Form — Name, Email, Phone, Branch, Message + Turnstile

2-Step form:

  • Step 1 — Client Details: name, company, title, phone, email, branch, account status
  • Step 2 — Project Details: project name/timing/location, plant, work times, site type, TMP needs, cost type, meeting preferences, optional file upload (5 MB max, DOCX/JPEG/PDF/PNG)

Special: Wellington-only "Lump Sum" cost type. Multiple conditional fields.

Location Pages ×5 Server

Wellington, Christchurch, Nelson, Blenheim, Timaru — all use the shared LocationPage component with city-specific props. Each page exports its own metadata and LocalBusiness JSON-LD.

Sections: Hero → Services list → Coverage Area → Why Choose Us → CTA

Privacy (/privacy) & Terms (/terms) Server

Dark hero + orange-rule content blocks in a 760px column. 7 privacy sections / 8 terms sections. Last updated March 2026.

Section 10

Security Implementation

Three-Layer Protection (All Forms)

Layer 1: Cloudflare Turnstile (invisible CAPTCHA)
Layer 2: Honeypot field ("companyPhone")
Layer 3: In-memory rate limiting (5 req / IP / 60s)

Cloudflare Turnstile

// Client — invisible, no UX disruption
<Turnstile sitekey={process.env.NEXT_PUBLIC_TURNSTILE_SITE_KEY!}
           size="invisible" onVerify={token => setTurnstileToken(token)} />

// Server — verifies token before processing
const res = await fetch('https://challenges.cloudflare.com/turnstile/v0/siteverify', {
  method: 'POST',
  body: JSON.stringify({ secret: process.env.TURNSTILE_SECRET_KEY, response: token }),
})

Honeypot Field

Hidden input name="companyPhone" positioned off-screen (left: -9999px), tabIndex="-1". If populated, server silently returns { success: true } — no signal given to attackers.

Rate Limiting

In-memory Map keyed by IP. Window: 60s, limit: 5 requests. IP extracted from x-forwarded-for → x-real-ip → 'unknown'. Returns HTTP 429 when exceeded.

Server-Side Validation

All inputs re-validated server-side regardless of client state. Email regex, phone regex (/^[0-9+\-\s]{7,20}$/), message length, file MIME type + extension + size (≤5 MB).

File Upload Security

Allowed MIME types: docx | jpeg | pdf | png. Extensions double-checked against MIME. Size capped at 5,242,880 bytes. Transmitted as Base64 JSON — never written to filesystem.

Section 11

Performance Optimisation

Next.js Image Optimisation

  • next.config.ts is empty — unoptimized flag removed, full Vercel optimisation active
  • next/image used for: logo (navbar/footer), leadership photos, group entity logos
  • priority flag on navbar logo (LCP element)
  • fill prop + sizes queries on leadership grid images:
    sizes="(max-width:640px) 100vw, (max-width:768px) 50vw, (max-width:1280px) 33vw, 260px"

Hero Image LCP

Hero uses CSS background-image (not <Image>), so LCP is optimised via:

<link rel="preload" as="image" href="/images/hero.webp" />

Added as first element in <head> in app/layout.tsx.

WebP Usage

All photographic assets are WebP: hero, worksite divider, leadership photos, service cards, role cards, group logos. SVG used for illustrations and maps.

Scroll Animations

RevealObserver.tsx uses IntersectionObserver — zero-cost until elements enter viewport. Above-fold elements revealed immediately on mount via requestAnimationFrame.

Animation Rules

Only transform and opacity animated (GPU-accelerated, no layout thrash). Never transition-all. Passive scroll listeners + RAF throttling on parallax.

Bundle Size

  • No large unused dependencies (puppeteer removed)
  • Lucide React — tree-shakeable icon library
  • Service card backgrounds loaded as CSS — no JS bundle impact
  • Font loading: preconnect + display=swap
Section 12

Accessibility Implementation

Semantic HTML

  • Every <section> has a descriptive aria-label
  • Heading hierarchy enforced: one H1 per page, H2 per section, H3 for sub-items
  • <ul>/<li> for all navigation and item lists
  • Decorative SVGs and icons use aria-hidden="true"

Forms

  • Every input paired with <label htmlFor={id}>
  • Error messages use role="alert" and aria-live="assertive"
  • SelectWrapper: aria-haspopup="listbox", aria-expanded, Arrow/Enter/Escape keyboard support
  • Branch map pins: role="button", tabIndex={0}, onKeyDown handler

Focus States

CSS focus-visible on inputs: border-color: var(--orange) + orange glow shadow. All buttons have visible focus rings.

Colour Contrast

Primary text: #ffffff on dark backgrounds. Secondary text: #7A8FA3. Orange #fd4f00 used only for accents and icons — not as body text on dark backgrounds.

Deployment Workflow

Local Development

npm install           # Install all dependencies
npm run dev           # Dev server at http://localhost:3000
npm run build         # Production build (check for TypeScript errors)
npm run lint          # ESLint via next lint

Rules: Never run multiple dev servers. Never use serve.mjs or custom static servers.

Version Control

git add <specific files>    # Stage named files only (never git add -A)
git commit -m "..."         # Include Co-Authored-By Claude line
git push                    # Push to origin/main (triggers Vercel deploy)

Rules: Do not push until explicitly instructed. Do not auto-commit.

Deployment

  • Platform: Vercel (auto-deploys on push to main)
  • Image optimisation: Handled by Vercel at no additional cost
  • Environment variables: Set in Vercel project dashboard

Section 18

Final Summary

9/10
Template Reusability Score

The primary conversion work is content extraction — replacing hardcoded company data with config variables — rather than architectural change. The form system, email pipeline, SEO structure, design system, and animation infrastructure are all immediately reusable without modification.

What Makes This Project Technically Strong
